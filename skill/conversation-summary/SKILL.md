# 对话上下文总结

## 用途

生成或更新一份简短的中文 Markdown 交接卡，保留当前任务背景、已确认结论、待解决问题和下一步动作，方便在其他 AI 工具或新对话中继续推进。输入可以是当前可见对话，也可以是公开可访问的对话分享链接。

## 默认输出

默认直接在对话中回复交接卡。只有当用户明确提供目标路径或要求输出到文件时，才写入或更新文件。

除非用户明确要求其他语言，否则始终使用中文输出。内容应保持平台中立，不绑定 ChatGPT、Claude、Codex、Cursor 或其他具体工具；只有当某个平台本身是必须保留的任务上下文时，才写入平台名称。

## 必须使用的格式

使用以下固定顶层结构：

```markdown
# 上下文交接卡

## 1. 当前任务

## 2. 当前目标

## 3. 已确定结论

## 4. 当前基准稿

## 5. 待解决问题

## 6. 下一步建议

## 7. 后续对话使用方式
```

不要添加额外的顶层章节。各章节内部可以按需使用项目符号。

## 根据分享链接生成交接卡

当用户提供公开对话分享链接时，先使用当前环境可用的网络、浏览器或 HTTP 工具打开链接并读取实际页面内容，再生成交接卡。常见链接形态包括但不限于：

- `chatgpt.com/share/...`
- `qianwen.com/share/chat/...`
- `doubao.com/thread/...`

处理分享链接时：

- 读取顺序：
  1. 先尝试当前可用的 HTTP、静态抓取或 `web_fetch` 类工具。
  2. 如果当前环境允许执行本 skill 的脚本，必须在任何浏览器渲染之前运行 `python3 scripts/render_share_text.py <URL> --static-only`。即使 HTTP 只读到少量标题、空容器或疑似动态页面，也要先跑脚本；脚本会完整读取响应并做静态 HTML、SSR/路由数据、嵌入 JSON、长文本字段和已知公开分享 API 抽取，不会自动启动系统浏览器，也不会自动安装 Playwright 浏览器引擎。
  3. 如果分享对话很长、轮次很多或主题分散，改用 `python3 scripts/render_share_text.py <URL> --static-only --compact`。对 `qianwen.com/share/chat/...` 这类接口型分享页，优先使用 `--compact`。`--compact` 会优先列出所有用户轮次和每轮回答摘录，避免模型只总结尾部内容。
  4. 如果 `--static-only` 输出了对话正文或长文本，直接基于该输出总结，不要再尝试浏览器渲染、Playwright 或安装浏览器引擎。
  5. 只有在 `--static-only` 明确失败，且确实需要本机浏览器渲染时，才继续尝试渲染型浏览器工具，例如 Browser、Computer Use、GUI 浏览器或其他可执行 JavaScript 的页面访问方式。
  6. 如果需要本机浏览器渲染，优先让用户或调用方显式设置 `CONVERSATION_SUMMARY_BROWSER` 为 Chromium 系浏览器可执行文件路径后重试；或在确认可接受启动系统浏览器后使用 `--allow-system-browser` 自动发现浏览器。这对 macOS、Windows 和信创 Linux 环境尤其重要，可避免误启动 GUI 浏览器并触发系统崩溃弹窗。
  7. Playwright 仅作为显式 opt-in 的最后兜底；不要自动执行 `playwright install`，也不要默认启动 Playwright 托管浏览器。只有用户明确允许安装依赖或浏览器引擎时，才建议安装后重试。
  8. 从读取到的脚本文本、API 数据、渲染后可见文本、可访问性树、页面正文或复制出来的页面内容中提取对话。
  9. 只有静态抓取、bundled script、渲染型浏览器、系统浏览器脚本和已安装 Playwright 能力都不可用或都失败后，才说明当前工具无法直接读取分享页内容，并请用户粘贴对话正文或页面可见内容。
- 基于实际读取到的对话内容总结，不要只总结链接本身。
- 提取用户消息、AI 回复、任务目标、用户确认的结论、当前基准稿、待解决问题和下一步建议。
- 忽略页面导航、按钮、登录提示、推荐内容、广告、页脚和其他非对话噪声。
- 不要在 skill 中假设固定页面结构、DOM 选择器、JSON 字段或平台专用解析规则；页面结构可能变化，应以实际可读取内容为准。
- 静态页面正文可能不在可见 HTML 中，而是嵌在 `<script>` 的 SSR 数据、路由数据或 JSON 字符串里；在判定需要浏览器前，应先尝试通用嵌入文本抽取。
- 有些分享页的首屏 HTML 不含正文，但前端会调用公开分享 API 返回对话数据；这仍属于非浏览器静态读取路径，应优先由 bundled script 处理。例如 `qianwen.com/share/chat/...` 可通过其公开分享接口读取 `record_list`。
- 对 `qianwen.com/share/chat/...`，不要在只看到“静态抓取 24 字符”后转向浏览器；必须先运行 `python3 scripts/render_share_text.py <URL> --static-only`，该脚本会尝试公开 `share/info` 接口。
- 不要用 `curl | head`、`head -c`、只 grep `_SSR_DATA` 等片段检查来判定静态抽取失败；很多分享页会把正文放在响应后半段、流式脚本或非 `_SSR_DATA` 的长 JSON 字符串中。应运行 bundled script 完整抽取。
- 不要仅凭 HTTP、静态抓取或 `web_fetch` 失败就判断链接需要登录、Cookie、会话状态或不是公开链接。除非渲染型浏览器实际显示登录墙、权限提示或不可访问错误，否则只说明当前工具链未能读取到内容。
- 如果用户确认链接在无痕浏览器可直接访问，应把它视为公开页面；在有 bundled script 的环境中，优先尝试 `--static-only`，再考虑浏览器读取路径。
- 如果当前环境没有 Browser、Computer Use、Playwright 等浏览器工具，但有 shell 或脚本执行能力，应优先尝试 bundled script，而不是直接要求用户粘贴正文。脚本默认不自动启动本机浏览器；只有设置 `CONVERSATION_SUMMARY_BROWSER` 或传入 `--allow-system-browser` 时，才会复用 macOS、Windows、Linux/信创系统中已有的 Chrome、Edge、Chromium、Brave，并使用临时 profile，避免污染用户日常浏览器配置、Cookie、历史记录和插件状态。
- 如果机器上没有任何可用浏览器引擎，无法可靠读取必须执行 JavaScript 后才显示内容的页面；这属于工具链能力不足，不代表链接不是公开链接。
- 如果链接无法访问、内容为空、页面动态渲染失败或当前工具无法读取，不要臆测链接中的对话内容。说明已尝试的工具类别和无法读取的具体表现，并请用户粘贴对话正文或页面可见内容。
- 如果只读取到部分内容，在交接卡相关章节标注 `仅基于可读取部分` 或 `不确定`。
- 对长对话生成交接卡时，不要只基于最后几轮总结；必须先看 `User Turn Index` 或轮次标题，先归纳 2-6 个主要主题簇，再写交接卡，确保 `当前任务`、`当前目标`、`已确定结论`、`当前基准稿`、`待解决问题` 覆盖所有主要主题簇。
- 如果 `User Turn Index` 显示多个不相关主题，不要把最后一个主题当成整段对话的唯一任务。应在 `## 1. 当前任务` 中说明这是一次多主题咨询总结，并列出主题范围；在 `## 3. 已确定结论` 中按主题分组保留结论。

## 总结规则

- 保持交接卡简短且高信号。保留决策、约束、当前状态和可复用的工作内容；不要复述完整聊天记录。
- 优先保留用户明确接受、要求、修正或确认的结论。弱化模型的发散建议、已拒绝方案和探索性分支。
- 信息缺失时标注 `不确定`，不要编造细节。
- 如果结论冲突且无法从可见对话中判断，以 `**冲突结论：**` 的形式在相关章节单独列出。
- 在 `## 4. 当前基准稿` 中保留可供下一轮继续迭代的当前版本、模板、方案、草稿、代码形态或关键内容。如果还没有具体基准稿，写 `不确定`，或用一句话说明最新可用状态。
- 在 `## 6. 下一步建议` 中只给 1-3 个具体下一步动作，避免大而全的路线图。
- 在 `## 7. 后续对话使用方式` 中说明新对话应如何粘贴或提供这份交接卡，让下一个 AI 能继续工作。

## 更新已有交接卡

当用户提供已有交接卡并要求更新时：

- 以用户新确认的信息为准，覆盖旧交接卡中的过时内容。
- 删除已失效或已被替代的细节，不要堆积历史记录。
- 保留仍然有效的结论和最新可复用基准稿。
- 如果新旧信息冲突且无法明确判断，保留双方并归入 `**冲突结论：**`。

## 质量检查

回复前检查：

- 输出以 `# 上下文交接卡` 开头。
- 七个必需章节完整且顺序正确。
- 交接卡简短，并聚焦上下文迁移。
- 不确定或缺失的信息已标注 `不确定`。
- 下一步建议不超过三个动作。
