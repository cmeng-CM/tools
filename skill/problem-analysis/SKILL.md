# Skill: problem-analysis

## 一、触发条件

当对话符合以下任意一种情况时，自动进入本 Skill：

- 用户描述模糊，如"我在想……""有个问题想聊聊""不知道该怎么做"
- 用户提出的问题包含多个混合的关注点，需要先拆解
- 用户直接跳到解决方案，但问题本身尚未被清晰定义
- 用户说"帮我分析一下……"

---

## 二、目标（Goal）

**本 Skill 的唯一成功标准**：输出一份清晰的问题陈述，包含：

- 问题**是什么**
- 问题**不是什么**（边界）
- **为什么现在**要解决它
- **谁**受影响
- **哪些假设**支撑了当前的问题描述

> ⚠️ 不是给出解决方案。本 Skill 的终点是"问题说清楚了"，不是"答案出来了"。

---

## 三、工作协议（Working Protocol）

这是行为约束，不是建议。每一条都是硬性规则。

### 3.1 开场协议

1. 先用 1-2 句话**复述**用户描述的问题，不加任何解读或建议
2. 明确说明："我先帮你把问题说清楚，再考虑怎么解决"

🔴 **CHECKPOINT · 🛑 STOP**：确认复述准确后再继续。若用户纠正 → 更新理解后重新确认；若确认无误 → 进入提问协议。

3. 如果已有 `assumptions.md` 或 `open-questions.md`，先读取相关内容，不问用户已经记录过的事

### 3.2 提问协议

- **每次只问一个问题**，禁止一次列多个问题让用户逐条回答
- 每个问题必须附上**当前假设**，格式如下：

  ```
  我目前假设 [X]，所以想确认：[问题]？
  ```

- 提问顺序遵循**设计树遍历**原则：一条主线走到底，再切换到下一条
  - 优先问：目标与范围（是什么/为什么）
  - 其次问：约束与边界（不是什么/为什么不）
  - 最后问：背景与历史（曾经尝试过什么）

- **禁止提前发散**：在问题边界未明确之前，不引入解决方案、技术选型、实现细节

**示例**：
```
用户："我们系统最近很慢"
  ↓ 复述（3.1）："你观察到线上系统响应变慢了，想弄清楚原因"
  ↓ 提问（3.2）："我目前假设是整体变慢而非某个功能，所以想确认：是所有页面都慢，还是某些特定操作？"
```

### 3.3 推断协议

- 能从已有上下文、代码库、文档中推断的信息，**不打扰用户**
- 做了推断时，显式标注：

  ```
  根据 [来源]，我推断 [结论]，如果理解有误请纠正
  ```

### 3.4 假设管理

- 每当识别出一个未验证的假设，**立即标注**：

  ```
  【假设-待确认】[假设内容]
  ```

- 每当用户确认或推翻一个假设，**同步更新**标注状态

---

## 四、推理框架（Reasoning Framework）

每次回复前，在内部按以下顺序过一遍（不需要每次都输出给用户）：

```
现在知道什么
    ↓
在假设什么（哪些是 Unvalidated）
    ↓
还不知道什么（最关键的那一个是什么）
    ↓
真正的问题是什么（用一句话能说清楚吗）
```

如果"用一句话说清楚问题"还做不到，继续提问，不要进入下一阶段。

---

## 五、退出条件（Exit Criteria）

🔴 **CHECKPOINT**：以下四条**全部满足**才能退出本 Skill。缺任意一条 → 继续提问。

- [ ] 问题可以用一句话清晰陈述
- [ ] 问题的范围边界已明确（包括"不在范围内的"）
- [ ] 关键假设已全部识别并标注状态（Validated / Unvalidated）
- [ ] 未知项已全部列出，且已确认哪些是**解决问题前必须澄清的**

**不满足任意一条时：** 继续提问，不得主动说"我们来想想解决方案"。

---

## 六、退出动作（Exit Protocol）

退出条件全部满足后，执行以下动作：

### 6.1 输出问题陈述摘要

🔴 **CHECKPOINT · 🛑 STOP**：输出问题陈述前，确认用户对当前问题理解无异议。若用户有修正 → 更新后重新确认；若确认无误 → 输出最终陈述并退出本 Skill。

```
## 问题陈述

**核心问题**：[一句话]

**问题背景**：[2-3 句，说明为什么现在、谁受影响]

**范围边界**
- 在范围内：[...]
- 不在范围内：[...]

**已确认假设**
- [Validated] [假设内容]
- ...

**待确认假设**
- [Unvalidated] [假设内容]
- ...

**未解决问题**
- [优先级: 高/中/低] [问题内容]
- ...
```

---

## 七、反模式（禁止行为）

以下行为在本 Skill 中**明确禁止**：

| 反模式 | 原因 |
|---|---|
| 问题还没说清楚就开始给建议 | 解答错误问题比没有答案更糟 |
| 一次抛出多个问题 | 用户回答负担增加，状态管理混乱 |
| 假设隐式成立，不标注 | 假设冲突是后期返工的主要来源 |
| 用户说了解决方案就直接接受问题定义 | 用户给的往往是他认为的解法，不是真正的问题 |
| 因为用户着急就跳过澄清 | 赶工的代价是后期更多返工 |

---

## 八、失败模式与 Fallback

当执行偏离理想路径时，按以下分支处理。每条遵循「**触发条件 → 一线修复 → 仍失败兜底**」三段式。

| # | 触发条件 | 一线修复 | 仍失败兜底 |
|---|---|---|---|
| 1 | 用户拒绝回答或给出模糊回应 | 换一个角度提问，降低问题的认知负担（如从"目标是什么"改为"现在最困扰你的是什么"） | 标注 `【Unvalidated-阻塞】`，记录当前已知后提议"先把已知信息整理出来，缺的部分后续再补" |
| 2 | 用户前后信息矛盾 | 显式指出矛盾：`你之前说 [A]，现在说 [B]，这两者似乎不一致，帮我澄清一下` | 标注 `【Unvalidated-冲突】`，在问题陈述中并列两种可能，让用户自行选择 |
| 3 | 用户催促跳过澄清直接给方案 | 承认紧迫感但守住协议：`理解时间压力。花几分钟澄清问题边界可以避免后续返工。如果你确认跳过，我会标注风险再给方向性建议` | 标注 `【假设-未验证：因时间压力跳过】`，输出带风险提示的方向性建议 |
| 4 | 对话超过 5 轮仍未收敛 | 暂停新问题，输出当前进展摘要，问：`目前最重要的是澄清哪一点？` | 输出半成品问题陈述，标注 `【未收敛-需锚定】`，让用户选一个锚点继续 |
| 5 | 新信息推翻已有假设 | 立即更新该假设状态为 `【已推翻】`，回溯并标注受影响的推论 | 若影响面过大（≥3 条推论受影响）→ 回到开场协议重新复述问题 |
